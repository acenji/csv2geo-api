# Contributing to CSV2GEO API

Thank you for considering contributing to the CSV2GEO API documentation and SDKs! We welcome contributions from the community.

## Ways to Contribute

- **Report Bugs** - Found an issue? Open a GitHub issue
- **Improve Documentation** - Fix typos, clarify explanations, add examples
- **Add Examples** - Share code samples in any language
- **Build SDKs** - Help create or improve client libraries
- **Suggest Features** - Have an idea? We'd love to hear it

## Getting Started

### 1. Fork the Repository

Click the "Fork" button on GitHub to create your own copy.

### 2. Clone Your Fork

```bash
git clone https://github.com/YOUR_USERNAME/csv2geo-api.git
cd csv2geo-api
```

### 3. Create a Branch

```bash
git checkout -b feature/your-feature-name
```

### 4. Make Your Changes

Edit files, add examples, improve docs, etc.

### 5. Test Your Changes

- For OpenAPI spec: Validate at [editor.swagger.io](https://editor.swagger.io)
- For docs: Preview locally if using a docs framework
- For SDKs: Run the test suite

### 6. Commit Your Changes

```bash
git add .
git commit -m "Add: description of your changes"
```

**Commit message prefixes:**
- `Add:` - New feature or content
- `Fix:` - Bug fix
- `Update:` - Updates to existing content
- `Docs:` - Documentation only changes
- `Refactor:` - Code restructuring

### 7. Push and Create a Pull Request

```bash
git push origin feature/your-feature-name
```

Then open a Pull Request on GitHub.

## Code Style

### OpenAPI Specification
- Use OpenAPI 3.0.x format
- Include descriptions for all endpoints and parameters
- Provide realistic examples
- Validate with Swagger Editor before submitting

### Documentation (Markdown)
- Use clear, concise language
- Include code examples where helpful
- Test all code snippets
- Use proper heading hierarchy (h1 → h2 → h3)

### Code Examples
- **Python**: Follow PEP 8, use type hints
- **JavaScript/Node.js**: Use modern ES6+ syntax
- **PHP**: Follow PSR-12
- **Go**: Run `gofmt`
- All: Include error handling, comments for complex logic

## Pull Request Guidelines

1. **One feature per PR** - Keep PRs focused
2. **Update the changelog** - Add your changes to CHANGELOG.md
3. **Write clear descriptions** - Explain what and why
4. **Reference issues** - Link to related issues if applicable
5. **Be patient** - We review PRs as quickly as possible

## Reporting Issues

When reporting bugs, please include:

- Clear description of the issue
- Steps to reproduce
- Expected vs actual behavior
- Your environment (OS, language version, etc.)
- Relevant code snippets or error messages

## Feature Requests

We love hearing ideas! When suggesting features:

- Check existing issues first
- Describe the use case
- Explain how it would help users
- Consider if it fits the project scope

## Code of Conduct

- Be respectful and inclusive
- Welcome newcomers
- Give constructive feedback
- Focus on what's best for the community
